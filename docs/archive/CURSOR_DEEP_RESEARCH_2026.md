## 0. Словарь (актуальная линейка 3.x)

| Термин | Что это |
|--------|---------|
| **Agent** | Режим с tool-calling: чтение/правка файлов, терминал, поиск, браузер, MCP. |
| **Composer / Composer 2.5** | Модель Cursor для агентной работы; отдельный пул usage (Auto + Composer). |
| **Agents Window** | Отдельное окно агентов (3.0+): параллельные табы, worktree, `/multitask`. |
| **Checkpoints** | Локальные снимки файлов **внутри сессии Agent** (не Git). |
| **Max Mode** | Расширение контекста до максимума модели (дороже по токенам). |

## 1. Режим Agent (Cursor Agent / Composer)

### 1.1. Файлы: что может делать автономно за одну сессию

**Официально (жёсткие факты):**

| Возможность | Лимит |
|-------------|-------|
| Вызовы tools (read/edit/search/terminal/…) | **Нет лимита** на число вызовов за задачу |
| Редактирование | Любые файлы в workspace, кроме защищённых настройками |
| Checkpoints | Автоматически **перед значимыми правками**; откат по таймлайну чата |
| Индексация для поиска | Все файлы, кроме `.gitignore` / `.cursorignore`; sync ~каждые **5 мин** |

**Практический потолок (не документирован Cursor, эмпирика 2025–2026):**

| Метрика | Рабочий регламент для соло |
|---------|----------------------------|
| Файлов, осмысленно затронутых за **одну** сессию | **≤ 8–12** (больше → дрейф архитектуры) |
| Строк кода, прочитанных агентом подряд без субагента | **~3 000–8 000** полезных LOC в контексте; остальное — через grep/semantic search |
| Параллельные «большие» задачи | **1 активный Agent-чат = 1 фича**; вторую — новый чат или worktree |
| Длительность непрерывной автономии | **15–40 мин** типично; длинные прогоны (60+ мин) требуют узкой постановки и контрольных точек |

**Защиты файловой системы (Settings → Agents → Protection):**

- **File-Deletion Protection** — блок авто-удаления.
- **Dotfile Protection** — блок авто-правки `.gitignore` и др. dotfiles.
- **External-File Protection** — блок записи **вне** workspace.
- Каталог **`.cursor/`** (rules, commands, skills) — защищён sandbox’ом.

**Регламент оператора:**

1. Одна сессия Agent = **один артефакт**: «эндпоинт webhook X», «миграция Y», «фикс парсера Z».
2. Перед «большим рефакторингом» — **checkpoint** или `git commit` (см. §3).
3. Не давать задачу вида «перепиши весь `src/`» без списка **запрещённых** файлов.

### 1.2. Терминал

**Поведение:**

- Agent выполняет shell в **встроенном терминале**; читает stdout/stderr (в т.ч. через `@Terminals`).
- По умолчанию (3.5+): режим **Allowlist (with Sandbox)** — команды вне allowlist идут в **sandbox** с авто-запуском; эскалация — по запросу.

**Sandbox (ограничения по умолчанию):**

| Ресурс | В sandbox |
|--------|-----------|
| Файлы | Read вне workspace; **R/W только workspace** (+ настраиваемые пути в `sandbox.json`) |
| Сеть | **Заблокирована**, пока не включена политика / allowlist доменов |
| `/tmp` | Полный доступ |
| `.cursor/` | Защищён |

**Windows:** sandbox в **WSL2**; без WSL2 — fallback: **ручное подтверждение** каждой команды.

**Auto-run режимы:**

| Режим | Когда использовать соло-оператору |
|-------|-----------------------------------|
| **Allowlist (with Sandbox)** | **Дефолт:** `pytest`, `ruff`, `python -m …` в sandbox; `pip install` — с сетью через `sandbox.json` |
| **Allowlist** | Только явно разрешённые команды без sandbox |
| **Run Everything** | **Запрещено** для прод-данных и `.env` |

**Что агенту фактически «запрещено» без вашего Run / allowlist:**

- Сеть (установка пакетов, `curl` API, деплой) — в sandbox.
- Запись вне workspace.
- Модификация dotfiles (если включена Dotfile Protection).
- Удаление файлов (если File-Deletion Protection).
- Команды с побочными эффектами: `git push --force`, `DROP DATABASE`, `rm -rf`, правка production secrets — **всегда** требуют осознанного подтверждения; allowlist **не** считается security boundary (enterprise docs).

**Регламент команд для webhook/FastAPI стека:**

```text
Разрешить в allowlist (пример):
  python -m pytest
  python -m ruff
  uvicorn (только dev, без prod URL)

Всегда вручную:
  git push
  alembic upgrade (на prod DSN)
  любая команда с реальным TELEGRAM_BOT_TOKEN / DATABASE_URL из .env
```

**`sandbox.json` (репо):** для Neon/pip — явный `networkPolicy.allow`: `pypi.org`, `files.pythonhosted.org`, `*.neon.tech` (по необходимости). Шаблон — [cursor.com/docs/reference/sandbox](https://cursor.com/docs/reference/sandbox).

### 1.3. Лимиты контекста и деградация

**Официально:**

- Базовое окно большинства моделей: **~200k токенов**.
- **Max Mode** — до максимума модели (например **1M** у Claude 4.5/4.6 Sonnet, GPT-5.4/5.5 в Max); тарификация по API-ставке модели.
- В UI: **индикатор заполнения контекста** в строке ввода — ориентир для сброса чата.

**Практические цифры (консервативный регламент):**

| Сигнал | Порог | Действие |
|--------|-------|----------|
| Индикатор контекста | **> 60–70%** | Новый чат + `@` только нужные файлы |
| Сообщений в одном чате | **> 25–40** с правками | Новый чат; перенести вывод в `docs/team/common/STATUS.md` |
| Один файл в контексте | **> 400–600 строк** | Дробить модуль или `@` конкретные функции через правило «не трогать» |
| «Забывание» | Агент предлагает откатить уже согласованное API/имена | **Стоп** → commit → новый чат с `@ARCHITECTURE` / rule |

**Оценка токенов (грубо):**

| Объём | Токены (оценка) |
|-------|-----------------|
| 300 строк кода | 2 000–4 000 |
| 10 сообщений Composer + авто-контекст (табы, индекс) | 30 000–60 000 |
| До «полезного» остатка в 200k окне | часто **30–50%** окна уже занято **до** вашего промпта |

**Когда включать Max Mode:** рефакторинг **> 15 файлов** или трассировка сквозь весь `src/`; иначе — обычное окно + узкий `@`.

**Subagents (Explore):** широкий поиск выносится в отдельное окно — **снижает засорение** основного чата; явно: «use a subagent to find…».

## 2. Сложный контекст (Context Management)

### 2.1. Правила вместо `.cursorrules`

**Статус 2026:** `.cursorrules` в корне — **legacy**, миграция в `.cursor/rules/*.mdc`.

**Приоритет слияния:**

1. Team Rules (Enterprise)  
2. **Project** `.cursor/rules/`  
3. User Rules (глобально)  
4. Legacy `.cursorrules`  
5. `AGENTS.md` (упрощённый вариант)

**Типы активации (frontmatter `.mdc`):**

| Тип | `alwaysApply` | `globs` | `description` | Когда |
|-----|---------------|---------|---------------|-------|
| Always | `true` | — | — | ≤ **1–2** файла, < **80 строк** суммарно |
| Auto file | `false` | `src/**/*.py` | — | Стиль/паттерны для Python |
| Intelligent | `false` | — | да | Архитектура, «не трогать Telethon» |
| Manual | `false` | — | — | `@coder`, `@lead-architect` по роли |

**Регламент «не сносить рабочий код» (для вашего репо):**

```yaml
# .cursor/rules/guardrails.mdc (рекомендуемый шаблон)
---
description: Запреты и границы правок для Agent
alwaysApply: true
---

## Не изменять без явной задачи в docs/team/common/TASKS.md
- src/tg_client.py, src/tg_monitor.py (Telethon)
- docs/team/common/TASKS.md, docs/team/architect/TZ_API.md
- .env (только чтение; правки — в .env.example + инструкция владельцу)

## Обязательно перед правкой модуля
- Прочитать docs/team/architect/ARCHITECTURE.md
- Одна задача = один модуль из списка «В работе» в TASKS

## Стиль правок
- Минимальный diff; не рефакторить соседние файлы
- После сессии: блок в docs/team/common/STATUS.md (Coder)
```

**Ошибки, ведущие к «сносу»:**

| Ошибка | Последствие |
|--------|-------------|
| `alwaysApply: true` на 500+ строк правил | Съедает контекст; модель игнорирует хвост |
| Противоречивые rules (Coder vs Lead) | Случайные правки TASKS/ops |
| Нет globs на «опасные» файлы | Agent трогает Telethon при задаче «парсер FL» |

### 2.2. @-меншны и внешняя документация

**Доступные @ (официально):**

| Меншн | Назначение |
|-------|------------|
| `@file` / `@folder` | Точечный код |
| `@Docs` | Индексированная дока (встроенная + Add new doc по URL) |
| `@Web` | Актуальная информация из сети |
| `@Terminals` | Логи ошибок |
| `@Commit` / `@Branch` | Diff |
| `@Past Chats` | Контекст прошлого чата |
| `@Browser` | UI (Agents Window) |
| `@rule-name` | Ручное правило из `.cursor/rules/` |

**Критично для галлюцинаций API:**

- **`@Docs` не остаётся в контексте навсегда** — на **каждое** сообщение с опорой на внешнее API нужно снова `@Docs` (или rule с `@` на выжимку).
- Большие сайты (2000+ страниц): в контекст попадает **релевантный фрагмент по запросу**, не весь сайт.
- Для **кривых/нестабильных API** — не полагаться на `@Docs` одного URL; сделать **`docs/ops/integrations/<name>.md`** (≤ 150 строк): эндпоинты, примеры запрос/ответ, коды ошибок, «что уже сломано у нас».

**Регламент подачи документации:**

```text
1. Выжимка в репо (docs/ops/...) — источник истины
2. @Docs — только как дополнение к выжимке
3. @Web — для changelog / breaking changes за последние месяцы
4. В промпте: «если поля нет в @Docs и docs/ops/... — НЕ выдумывать, спросить»
5. Один промпт = один внешний сервис (Telegram webhook ≠ Neon ≠ OpenRouter)
```

**`.cursorignore`:** исключить `data/`, `*.db`, `radar.log`, `.venv`, большие JSON — иначе шум в semantic search.

## 3. Модульность и итерационная разработка

### 3.1. Структура Python (FastAPI + Postgres/Neon) под Cursor

Цель: **изолированные чаты** = **изолированные границы импорта**.

**Рекомендуемое дерево (зелёное поле / webhook-сервис):**

```text
src/
  main.py              # только lifespan + mount routers
  config.py            # pydantic-settings, без бизнес-логики
  api/
    __init__.py
    health.py          # GET /health
    webhooks/
      telegram.py      # POST /webhook/telegram — один файл на сессию Agent
      payments.py      # при необходимости
  services/            # чистая логика, без FastAPI Request
    notify.py
    orders.py
  db/
    pool.py            # asyncpg / SQLAlchemy async — один модуль
    queries/
      leads.sql        # или *.py с сырыми SQL
  schemas/             # Pydantic request/response
  workers/             # фоновые задачи (опционально)
tests/
  test_webhooks_telegram.py
docs/
  team/architect/ARCHITECTURE.md # 1 страница — границы модулей
  ops/integrations/telegram_bot_api.md
```

**Правила модульности для Agent:**

| Правило | Число |
|---------|-------|
| Максимум зависимостей у одного `services/*.py` | **≤ 3** внутренних модуля |
| Размер файла «зоны ответственности» | **150–250 строк**; больше — делить до следующей сессии |
| Публичный контракт модуля | Только через `schemas/` + docstring в `api/` |
| Сессия Agent | Трогает **1 каталог** (`api/webhooks/` ИЛИ `services/` ИЛИ `db/`) |

**Связка с текущим репо (радар):** уже есть слоистость (`parsers`, `storage`, `filters`). При переходе на FastAPI — **не смешивать** Telethon и webhook в одном чате; отдельные TASKS и rules с globs.

**Neon:** DSN только в `.env`; Agent работает с **`docs/ops/NEON_CURSOR.md`** + миграции в `db/migrations/`; одна сессия = одна миграция.

### 3.2. Git + Checkpoints = защита от безвозвратного затирания

**Два уровня отката:**

| Уровень | Хранение | Назначение |
|---------|----------|------------|
| **Checkpoints** | Локально, сессия Agent | Быстрый откат «агент пошёл не туда» |
| **Git** | Репозиторий | Постоянная история, ветки, review |

**Регламент 100% восстановимости (оператор-начинающий):**

```text
ПЕРЕД каждой сессией Agent:
  git status          # чисто или осознанный WIP
  git checkout -b agent/YYYYMMDD-краткое-имя

КАЖДЫЕ 20–30 мин или после «зелёного» pytest:
  git add -p          # или add конкретных файлов
  git commit -m "agent: <что сделано>"

ПОСЛЕ успешной проверки:
  merge в main только вручную / через PR

НИКОГДА не полагаться только на Checkpoints для прод-кода
```

**Cursor 3.x дополнения:**

- `/worktree` — изолированная копия для эксперимента агента.
- `/best-of-n` — сравнение моделей; **не** для прод-ветки без merge-контроля.

**Связка с процессом репо:** Coder пишет в `STATUS.md`; Lead держит `TASKS.md` — Agent **не** меняет TASKS без `@lead` rule.

## 4. Технические слепые зоны

### 4.1. Где Agent систематически проваливается

| Класс задачи | Почему ломается | Что делать оператору |
|--------------|-----------------|----------------------|
| **Многошаговые OAuth / 3-legged API** без песочницы | Нет интерактивного браузера с вашей сессией | Ручной шаг «получи refresh token» → выжимка в `docs/ops/` |
| **Telethon / MTProto** (сессии, 2FA, flood-wait) | Долгий async state, ban risk | Отдельный чат; **не** «почини всё»; только `tg_smoke.py`-сценарии |
| **Webhook + идемпотентность + порядок событий** | Гонки, дубли update_id | Явные тесты с фикстурами JSON; Agent пишет тест **до** прод-логики |
| **Prod-only баги** (таймзоны, encoding, proxy) | Sandbox без вашей сети | `@Terminals` + реальный лог; не Run Everything |
| **Нестандартные UI / legacy фронт** | Нет стабильного DOM в контексте | Design Mode (Agents Window) или скрин + узкая задача |
| **Схема БД «на глаз»** | Галлюцинация колонок | `@` миграция или `neon` MCP + запрет DDL без diff review |
| **Секреты в коде** | Модель вставляет ключ в файл | `.env` в guardrails; pre-commit / grep secret |

---

### 4.2. Пять триггеров «каши» и стоп-сигналы

| # | Триггер | Симптом (заметить за 5 мин) | Стоп-действие |
|---|---------|----------------------------|---------------|
| 1 | **Scope creep в промпте** | «Заодно поправь…» → >5 файлов в diff | Ctrl+C / Stop → commit текущее → новый чат с узкой задачей |
| 2 | **Повторный рефакторинг без теста** | Переименования, «улучшение структуры», падающий pytest | Запрет слова «refactor»; только «исправь тест X» |
| 3 | **Контекст > 70%** | Неверные имена функций из начала чата | Новый чат + `@STATUS` + 1 файл |
| 4 | **Дублирование модулей** | `utils2.py`, `helpers_new.py`, второй клиент API | Rule: «расширяй существующий модуль»; откат checkpoint |
| 5 | **Run Everything + сеть** | `pip install` лишнего, правка `.env`, curl на неизвестные URL | Вернуть Allowlist (with Sandbox); rotate keys если светились |

**Красные флаги в diff (отклонить без review):**

- Удаление **> 30 строк** рабочего кода без TASKS.
- Изменение **> 3 файлов** вне объявленной задачи.
- Появление `except: pass`, закомментированных тестов, отключения RLS/SSL.

## 5. Методология соло-оператора (сводный регламент)

### 5.1. Один цикл работы (45–90 мин)

```text
1. Lead/владелец: одна строка в TASKS «В работе»
2. git checkout -b agent/...
3. Новый Agent-чат, модель: Composer 2.5 или Auto; Max Mode — только если >10 файлов
4. @TASKS @ARCHITECTURE @<один модуль>
5. Промпт: цель, критерий готовности, список НЕ ТРОГАТЬ
6. Agent работает; оператор только Run на опасных командах
7. python -m pytest (или шаг из TESTING.md)
8. git commit
9. STATUS.md — 5 строк «Сделано / Как проверить»
```

### 5.2. Выбор режима Cursor

| Задача | Режим |
|--------|-------|
| Одна функция, правка по месту | Inline edit / Cmd+K |
| Фича 2–8 файлов | Agent, один чат |
| Исследование «где в коде» | Ask / Agent + subagent explore |
| Параллельно: webhook + парсер | **Два таба** Agents Window или worktrees |
| Деплой на VPS | **Не** Agent «наугад»; чеклист в `docs/ops/`, ручной SSH |

### 5.3. Webhook-разработка (специфика)

```text
- Контракт: docs/ops/integrations/<provider>.md + fixtures/tests/fixtures/
- Agent: сначала test с payload из реального лога (@Terminals), потом handler
- ngrok/localtunnel: сеть в sandbox.json; токен webhook — только в .env
- Идемпотентность: явно в промпте «повторный update_id → 200 без побочных эффектов»
```

### 5.4. Связь с ИИ радара (OpenRouter)

| | Cursor Agent | `ai_analyze.py` |
|---|--------------|-----------------|
| Контекст | Весь репо (индекс) | **0** — один JSON на заказ |
| Память между вызовами | Есть (чат) | **Нет** |
| Риск | Портит код | Неверный вердикт |

Не смешивать: решения по заказам — OpenRouter; исполнение кода — Cursor с TASKS.
