# ТЗ — FL Radar (полное)

Версия: **1.2** · Lead

---

## 1. Суть продукта

**FL Radar** (фаза 0 в коде) — мониторинг **новых проектов на FL.ru и Kwork** по **твоим правилам** → уведомление в **Telegram** → **разбор ИИ** простым языком: сложность **для владельца с Cursor без навыков кода**, срок, деньги, черновик отклика.

**Развитие продукта** (TG-чаты, WP, аналитика): **`docs/team/PRODUCT_VISION.md`**, шаги — **`docs/ROADMAP.md`**.

Ты **сам** жмёшь «откликнуться» на FL. Авто-откликов нет.

---

## 2. Как настраивается выбор проектов

Подробно: **`docs/ops/FILTERS.md`**. Кратко:

| Уровень | Где настраиваешь | Что решает |
|---------|------------------|------------|
| **1. Лента FL** | `.env` → `FL_PROJECTS_URL` | Какая **категория** заказов (программирование, админка сайтов…) |
| **2. Слова** | `docs/ops/FILTERS.md` | **Тема** в заголовке/описании: берём / стоп (без кода) |
| **3. Бюджет** | `.env` → `MIN_BUDGET_RUB` | Не беспокоить, если слишком дёшево |
| **4. ИИ** | `docs/ops/PROFILE.md` + API-ключ в `.env` | Умный разбор **после** отсева; «брать / нет» и текст отклика |

**MVP (этап 1):** уровни 1–2 + короткое сообщение бота.  
**Этап 2:** уровень 3 + полный разбор ИИ в том же (или втором) сообщении.

**Не «выбираешь проекты галочками на FL»** — программа сама смотрит **всю ленту категории**, а ты задаёшь **правила**, что из неё тебе показывать.

## 3. Архитектура

Краткая схема ниже; **детальная** (слои, модули, Mermaid) — **`docs/team/ARCHITECTURE.md`**.

```
[ПК]
  → опрос FL.ru (URL из .env)
  → SQLite (уже виденные project_id)
  → фильтр слов (FILTERS.md)
  → [этап 2] загрузка страницы проекта + ИИ
  → Telegram Bot → ты
```

| Компонент | Решение |
|-----------|---------|
| Сервер | ПК владельца (MVP) |
| БД | SQLite `data/projects.db` |
| ИИ | Внешний API (OpenAI / Gemini / OpenRouter) — ключ в `.env`, опционально |
| Telegram | Bot API |

## 4. Этап 1 — MVP

### 4.1. Цикл
1. GET `FL_PROJECTS_URL` каждые `POLL_INTERVAL_MINUTES` (≥ 10).
2. Парсинг карточек: `project_id`, `title`, `budget_text`, `url`, `published_at`.
3. Новый `project_id` → SQLite.
4. Прошёл **FILTERS.md** → Telegram (коротко).
5. Лог `data/radar.log`.

### 4.2. Сообщение бота (этап 1 — без ИИ)

```
🆕 Новый проект на FL

[заголовок]

💰 [бюджет как на сайте]
🔗 [ссылка]

[кнопка: Открыть на FL]
```

### 4.3. Приёмка MVP
- [ ] Новый проект с ключевым словом → сообщение в TG.
- [ ] Без ключевого / со стоп-словом → в TG **нет**, в базе есть.
- [ ] Повтор того же `project_id` → без дубля.
- [ ] Настройка ленты через `.env`, слов — через `FILTERS.md`.

### 4.4. Код (Coder)
`config.py`, `fl_parser.py`, `storage.py`, `filters.py`, `telegram_notify.py`, `main.py` — см. TASKS.md.

## 5. Этап 2 — ИИ-разбор (целевой UX)

### 5.1. Когда вызывается ИИ
Только если проект:
- новый в БД;
- прошёл FILTERS + `MIN_BUDGET_RUB` (если задан);
- включён `AI_ENABLED=1` и есть `AI_API_KEY`.

Парсер **догружает полное описание** со страницы проекта (один GET на проект, не на весь каталог каждые 10 мин).

### 5.2. Вход для ИИ
- `title`, `budget_text`, `description` (полный текст с FL)
- `url`
- содержимое **`docs/ops/PROFILE.md`**: владелец **не программист**; сложность = реалистичность задачи **с Cursor**, не для разработчика

### 5.3. Что ИИ должен вернуть (структура)

| Поле | Для владельца | Пример |
|------|---------------|--------|
| `verdict` | Брать / Сомнительно / Пропустить | «Сомнительно» |
| **`work_summary`** | **Суть заказа** для TG (2–4 предложения, без жаргона) | «Нужен бот в Telegram под заявки…» |
| `difficulty` | Насколько **тебе с Cursor** (коротко, со словом Cursor) | «Легко — 1–2 вечера в Cursor» |
| **`approach`** | **Как сделаем** — 2 предложения: суть + **Cursor Agent** + вспомог. инструменты | «В Cursor соберём бота…; деплой через панель хостинга.» |
| `time_for_client` | Что **сказать заказчику** по сроку | «3–5 рабочих дней, если доступы сразу» |
| `money` | На бирже + **рынок** + **старт отклика** (шанс взять) | «На бирже: 3 000 ₽ \| Рынок: 12–18 тыс. \| Старт отклика: 14 000 ₽» |
| `reply_draft` | Отклик заказчику — **без ИИ/Cursor** в тексте | Живой русский, цена «от …» по «Старт отклика» |
| `risks` | 1–2 строки, на что смотреть | «Может попросить бесплатное ТЗ» |

Язык ответа: **простой, разговорный русский**, без жаргона и без «как ИИ».

### 5.4. Сообщение бота (этап 2 — полное)

```
🆕 Новый проект на FL

[заголовок]
📋 Задача:
[work_summary — или обрезка описания с биржи, если ИИ выключен]

💰 [бюджет с сайта]

———
🤖 Разбор

Вердикт: [Брать / Сомнительно / Пропустить]
Сложность (с Cursor, без кода): […]
Как сделаем: [approach — 2 предложения]
Срок заказчику: […]
Деньги: […]
Риски: […]

———
✍️ Черновик отклика (подправь под себя):

[reply_draft]

———
🔗 [url]
[кнопка: Открыть на FL]
```

Если `verdict` = **Пропустить** — можно **не слать** в TG (настройка `AI_NOTIFY_SKIP=0|1`, default 0 = не слать).

### 5.5. Конфиг `.env` (этап 2)

| Переменная | Описание |
|------------|----------|
| `AI_ENABLED` | 1 = включить разбор |
| `AI_API_KEY` | ключ провайдера |
| `AI_MODEL` | например `gpt-4o-mini` или модель Gemini |
| `AI_PROVIDER` | `openai` / `gemini` / `openrouter` |
| `MIN_BUDGET_RUB` | порог до ИИ |
| `AI_NOTIFY_SKIP` | 0 = не уведомлять о «Пропустить» |

### 5.6. Модуль (Coder)
`src/ai_analyze.py` — промпт, JSON-ответ, валидация полей, fallback: при ошибке API → короткое сообщение этапа 1 + «разбор ИИ недоступен».

### 5.7. Приёмка этапа 2
- [ ] Проект с подходящим описанием → TG с блоком «Разбор» и черновиком.
- [ ] `PROFILE.md`: оценка сложности под **Cursor / нулевой код** (WP-задача vs «с нуля платформа»).
- [ ] ИИ недоступен → всё равно приходит базовое уведомление.
- [ ] «Пропустить» не спамит (если `AI_NOTIFY_SKIP=0`).

### 5.8. Этика ИИ
- Не отправлять текст отклика на FL автоматически.
- Не хранить ключ API в Git.
- Ограничить длину промпта; не слать лишние PII в лог.

## 10. Бэклог (SaaS-трансформация)

**Статус:** не в работе у Coder. **Этапы 1–2 (локальный скрипт, SQLite, `.env`, `docs/*.md`) не меняются.** Coder сфокусирован только на **локальном MVP** по этому ТЗ; перенос на сервис — отдельная фаза после стабильного продукта «для себя».

### Оценка идеи (бот-помощник фрилансера с подпиской)

**Плюсы:** повторяемый продукт, подписка, каждый юзер со своими фильтрами и профилем, ИИ под персону — логичное развитие после того, как локальная версия докажет ценность.

**Риски и вопросы (проработать до кода SaaS):** условия использования FL.ru и массовый парсинг «для клиентов»; безопасность хранения **чужих** API-ключей и данных; биллинг, отмены, поддержка; нагрузка на FL (интервалы, fair use); конкуренция и дифференциация.

**Вывод:** вектор здравый, но **сначала** — рабочий локальный радар и этап 2 ИИ для одного владельца; SaaS — **отдельное ТЗ и архитектура**, не подмешивать в текущий `src/`.

### Тезисы бэклога (когда созреете)

1. **Бэкенд:** перенос логики на **FastAPI** и развёртывание на **VPS** (планировщик опроса FL, очереди, вебхуки Telegram при необходимости).
2. **Данные:** замена локальных **MD-файлов** на **Supabase** (или эквивалент): пользователи, подписки, ключевые слова, профили, история проектов / дедуп.
3. **Индивидуальные настройки в TG:** каждый пользователь **тонко настраивает, какие заказы показывать** (категории FL, белый/чёрный список, мин. бюджет, профиль для ИИ) — **через бота**, без правки файлов на диске; команды / inline / мини-сценарии.
4. **Мультитенант:** много пользователей на одном сервисе — изоляция `user_id`, свои ленты, дедуп, лимиты API ИИ.
5. **Динамический промпт:** ИИ под профиль и правила **конкретного** `user_id`, не один общий `PROFILE.md`.
*Детализация (тарифы, лимиты FL, модерация контента) — в отдельном документе после решения «идём в SaaS».*
