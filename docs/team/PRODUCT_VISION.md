# Видение продукта — поиск заказов (биржи + Telegram)

Версия: **0.2** · Lead · 2026-05-20

Для владельца простым языком — **`docs/FOR_YOU.md`**. Текущий рабочий код — **этап 0** (биржи на ПК). Ниже — **куда идём**.

---

## 1. Зачем

**Цель:** не «выиграть гонку на FL», а **находить заказы в Telegram** (и дублировать с бирж), **самому писать в личку** с предложением услуг — без автоспама и без риска для личного Telegram.

**Итог для владельца:** лиды в одном месте, разбор «стоит ли писать», черновик первого сообщения, позже — **сайт на WordPress** для настройки и **аналитика** (цены, спрос, что заходит).

---

## 2. Три роли Telegram (не один аккаунт)

| Роль | Аккаунт | Задача | Кто пишет |
|------|---------|--------|-----------|
| **Мониторинг** | Отдельный **купленный** номер | Telethon: читает чаты/каналы, куда этот аккаунт **вступил** | Никто — только чтение |
| **Рассылка / рабочий** | Второй **рабочий** номер | Позже: исходящие ЛС с оффером (если решите автоматизировать шаги) | **Владелец вручную** на старте — без массового спама |
| **Личный** | Твой основной | Получаешь **уведомления** (бот или канал), жмёшь «написать», отвечаешь сам | Ты |

**Важно:** покупной аккаунт **снижает риск для личного**, но **бан/ограничения возможны на мониторинговом** — нужны лимиты, паузы, немного чатов на старте, аккуратный FloodWait.

### Прокси и IP (обязательно)

**Мониторинговый (купленный) и личный аккаунт не должны сидеть на одном IP.**

Если купленный номер с **домашнего интернета** активно вступает в 10–20 фриланс-чатов, Telegram может связать сессии в одну «ферму» и ударить по **личному** (ограничения, теневой бан).

| Кто | Сеть |
|-----|------|
| **Мониторинг (Telethon)** | **Обязательно** чистые **приватные** прокси (SOCKS5 или HTTP). Отдельный IP на каждый мониторинговый аккаунт. В `.env` — отдельно от `TG_PROXY_URL` бота. |
| **Личный + бот-уведомитель (Bot API)** | **Домашний IP** (как сейчас `TG_PROXY_URL` только для Bot API — без смешивания с Telethon). |

**ТЗ для Coder (фаза 1+):** Telethon **всегда** через прокси; без прокси мониторинг **не запускать** (fail-fast в `config.py`). Прокси не коммитить в Git.

**Bot API** остаётся для: уведомлений тебе, паузы радара, кнопок — **не** для чтения чужих чатов без добавления бота.

## 4. Фазы (большой проект)

### Фаза 0 — биржи на ПК ✅ (сделано)

- **FL.ru** — парсер, до 3 страниц; **Kwork** — первая страница `wants`.
- SQLite `data/projects.db`, фильтр `FILTERS.md`, **`FILTER_WIDE`**, `MIN_BUDGET_RUB`.
- **ИИ** OpenRouter — разбор и черновик отклика на биржу.
- Уведомления и пауза — **Bot API** (`TG_PROXY_URL`), кнопки ⏸ ▶ ℹ.
- Запуск: `docs/ops/RUN.md`. Сводка модулей: **`docs/ROADMAP.md`**.

Владелец откликается **сам**; радар не пишет заказчикам. Биржи дальше — **дополнение** к TG.

### Фаза 1 — TG-мониторинг (MVP) ← **сейчас**

- Telethon на **мониторинговом** аккаунте **только через прокси** (§2).
- Список чатов (конфиг / позже WP).
- Фильтр стоп-слов → уведомление владельцу (источник, текст, ссылка на сообщение).
- Дедуп: одно сообщение не дважды, один заказ из нескольких чатов — по возможности схлопнуть.
- FloodWait / таймауты — не падать 24/7.

**Режим «Ночь» (антифрод):** фриланс-чаты ночью затихают; одинаковый опрос 24/7 с одной частотой похож на бота.

- В **`config.py`**: окно **«Ночь»** по умолчанию **02:00–07:00, Asia/Irkutsk** (настраиваемо в `.env`).
- Внутри окна: **увеличенные паузы** между проверками или **сон** (редкий poll / только heartbeat), чтобы имитировать живого пользователя, а не робота.
- Днём — обычный интервал; ночью — не «долбить» серверы TG с той же частотой.

### Фаза 2 — ИИ под «писать в ЛС»

- Вердикт: писать / сомнительно / пропустить.
- Черновик **первого личного сообщения** (без ИИ/Cursor в тексте для заказчика).
- Тот же профиль: не программист, делаешь через Cursor.

### Фаза 3 — WordPress «пульт»

- Сайт на WP: включить/выключить источники, чаты, фильтры, куда слать лиды.
- Запуск/настройка парсинга **с сайта** (бэкенд на VPS, WP — интерфейс).
- Не обязательно переносить парсеры в PHP — типично: **Python-сервис + API**, WP только UI.

### Фаза 4 — аналитика

- Все лиды в **Supabase**: источник, тема, бюджет (если есть), вердикт, исход (откликнул / нет — вручную или кнопкой).
- Отчёты: **ценообразование**, что в спросе, какие чаты дают конверсию.
- Основа для решений «куда усилить поиск».

### Фаза 5 — опционально

- SaaS / мультитенант — см. `docs/team/TZ.md` §10 (отдельно, не смешивать с фазой 0–2).

## 6. Что не делаем

- Авто-отклик на биржах.
- Массовая автоматическая рассылка в ЛС от имени владельца без его текста.
- Хранение паролей/сессий/прокси в Git.
- Смешивание SaaS и локального MVP в одном ТЗ без границ фаз.
- **Telethon и личный на одном IP** (см. §2).
- **История лидов в текстовых файлах** — только **Supabase (PostgreSQL)** с фазы 1+; аналитика и дедуп долгосрочно — там же.

### Данные и Supabase (жёстко)

В стеке закреплён **Supabase** — **база данных = основное хранилище** продукта (не «потом придумаем»).

| Что | Где |
|-----|-----|
| **Фаза 0 (сейчас)** | SQLite `data/projects.db` — **локальный временный кэш** бирж на ПК |
| **Фаза 1+** | Лиды, дедуп, настройки → **Postgres в облаке** (Neon Free; Supabase — лимит 2 проекта) |
| **Сессия Telethon** (`.session`) | **Не** как «главная копия на ПК», если деплой на VPS под WP/сервис. Coder сразу закладывает: сессия и секреты — **сервер + Supabase/секрет-хранилище**, не размазаны по ноутбуку и хостингу WP. WP **не** хранит `.session` — только Python-сервис. |
| **WordPress** | UI и отчёты через API; **не** замена БД и **не** хранилище session-файлов |

**Coder:** при старте фазы 1 читать этот блок до кода Telethon/миграций.
