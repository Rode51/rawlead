## 0. Северная звезда

**RawLead** — скрещение **dating-приложения** и **агрегатора вакансий/заказов**.

| Из dating | Из агрегатора |
|-----------|---------------|
| **Match** — «подходит / нет» одним жестом | FL.ru, Kwork, Telegram-чаты |
| **% совместимости** (профиль ↔ заказ) | Единая лента лидов |
| Карточка «ты ↔ возможность» | Фильтры, стоп-слова, ИИ-разбор |
| **Мэтч** = стоит откликнуться | Уведомление + черновик первого сообщения |

**Метафора UX:** не «ещё одна доска объявлений», а **Tinder для фриланса** — процент match, карточка лида, решение за секунды. На лендинге это уже видно: блок **источники → стрелка → карточка 88%** ([`design/wp/REFERENCE.md`](../design/wp/REFERENCE.md)).

| Слой | Что это |
|------|---------|
| **Сейчас (этап 0)** | Десктоп-радар + Telegram-бот: собираем лиды, считаем «берём/нет», обкатываем scoring |
| **v1 продукта** | Web / mobile с dating-UX поверх того же пайплайна (свайп, match %, история мэтчей) |
| **v2+** | WP-кабинет, аналитика, SaaS для других фрилансеров |

**Итог для пользователя:** меньше шума, больше **осмысленных совпадений** — как в dating, но про работу.

## 1. Зачем (этап 0)

**Цель:** не «выиграть гонку на FL», а **находить заказы в Telegram** (и дублировать с бирж), **самому писать в личку** — без автоспама и без риска для личного Telegram.

**Итог для владельца:** лиды в одном месте, **% совместимости и вердикт**, черновик первого сообщения; позже — **сайт / приложение** для настройки и **аналитика** (цены, спрос, конверсия мэтчей).

---

## 2. Три роли Telegram (не один аккаунт)

| Роль | Аккаунт | Задача | Кто пишет |
|------|---------|--------|-----------|
| **Мониторинг** | Отдельный **купленный** номер | Telethon: читает чаты/каналы, куда этот аккаунт **вступил** | Никто — только чтение |
| **Рассылка / рабочий** | Второй **рабочий** номер | Позже: исходящие ЛС с оффером (если решите автоматизировать шаги) | **Владелец вручную** на старте |
| **Личный** | Твой основной | Уведомления (бот), «написать», отвечаешь сам | Ты |

**Важно:** покупной аккаунт **снижает риск для личного**, но **бан возможен на мониторинговом** — лимиты, паузы, FloodWait.

### Прокси и IP (обязательно)

**Мониторинговый и личный аккаунт не должны сидеть на одном IP.**

| Кто | Сеть |
|-----|------|
| **Мониторинг (Telethon)** | **Обязательно** приватные прокси per acc. `TELETHON_PROXY_PROBE=1` — без живого прокси connect не идёт. |
| **Бот (Bot API)** | `TG_PROXY_URL` — отдельно от Telethon. **VPN на ПК** может резать доступ до прокси → бот молчит. |

## 4. Фазы (большой проект)

### Фаза 0 — радар на ПК ✅

- FL + Kwork + TG (3 acc), фильтр, ИИ, бот, пульт Tauri v2.
- Match-логика в боте: вердикт, бюджет, черновик (биржи).

### Фаза 1 — TG-мониторинг ✅ (MVP)

- Telethon через прокси, join-очередь, пересылка + разбор, ночные паузы.
- Proxy fail-closed + wait (не убивать бот при обрыве).

### Фаза 2 — ИИ и scoring «как dating»

- Явный **match %** в карточке (не только «Брать/Нет»).
- Черновик **первого ЛС** под профиль владельца.
- Дедуп: один заказ из нескольких чатов = один «кандидат».

### Фаза 3 — WordPress (маркетинг + учёба)

- ✅ Скелет страниц, **Kadence child**, референс E (editorial + bold type).
- Потом: кабинет на WP или API к Python-сервису ([`TZ_WP.md`](TZ_WP.md)).

### Фаза 4 — приложение (dating UX)

- Лента лидов: свайп / match / % совместимости.
- История мэтчей, «написал / пропустил», push.
- Backend: тот же пайплайн filter → AI → notify.

### Фаза 5 — аналитика

- Supabase/Neon: лиды, match rate, конверсия по источникам.
- Отчёты: цены, спрос, какие чаты дают мэтчи.

### Фаза 6 — опционально

- SaaS / мультитенант — [`TZ.md`](TZ.md) §10.


## 6. Что не делаем

- Авто-отклик на биржах.
- Массовая рассылка в ЛС без текста владельца.
- Секреты и `.session` в Git.
- Telethon и личный на одном IP.
- Dating-метафоры в спаме («ты мне подходишь» заказчикам) — только **внутри продукта** для владельца.
