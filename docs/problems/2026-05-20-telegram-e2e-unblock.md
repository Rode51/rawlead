# Telegram: довести до рабочих уведомлений (E2E с владельцем)

**Статус:** решено (2026-05-20)

## Симптом

- Радар крутится, FL парсится, в логе были **`chat not found`** и/или **`401 Unauthorized`** при `sendMessage`.
- Владелец уверен в своём **числовом id**; писал боту **`/start`**; возможны расхождения аккаунт/токен/revoke.
- Нужно **не отпускать** владельца к Lead, пока цепочка **токен → getMe → sendMessage в TELEGRAM_CHAT_ID** не даст **HTTP 200** с его стороны (проверка вместе в чате).

## Ожидалось

- Один понятный сценарий: из корня репо команда(и), которые владелец может запускать по инструкции **в этом тикете** (без копирования токена в чат).
- При типичных ошибках — **ясный текст** (уже частично сделано Mechanic в прошлом тикете; при необходимости усилить).
- По возможности — **минимальная** правка `src/` (например smoke-модуль или флаг `--telegram-smoke`), без нового бизнес-scope радара.

## Фактически

- Lead последний прогон API: см. историю в чате (были **401**, **400 chat not found**, **getUpdates: 0** до `/start`).
- Токен и `.env` в чат **не** выкладывать; проверки только у владельца локально или через команды, которые читают `.env` с диска.
- **Сейчас в коде:** скрипт `src/tg_smoke.py` и режим `python src/main.py --telegram-smoke` — один сценарий getMe → getUpdates → sendMessage с теми же переменными, что у радара.

## Чеклист владельца (до запуска команд)

1. **Один бот** — токен в `.env` (строка `TELEGRAM_BOT_TOKEN`) выпущен в @BotFather **для того бота**, с которым вы переписываетесь в Telegram.
2. **Без кавычек и мусора** — в `.env` значения без обрамляющих `"`/`'`; в конце строки нет пробела; нет двух разных токенов в разных копиях файла.
3. **`TELEGRAM_CHAT_ID`** — это **ваш** числовой id (личка) или id группы/канала, куда бот может писать; для лички это id **того же** пользователя, который открывает диалог с ботом.
4. **`/start`** — отправлен **этому** боту (по имени из getMe / из ссылки `t.me/...`) с **того аккаунта**, чей id в `TELEGRAM_CHAT_ID`.
5. Бот **не в чёрном списке** у вас (не «Заблокировать» в профиле чата).

### Если getMe ок, getUpdates = 0, sendMessage → «chat not found»

Это **не** ошибка токена. Нужно: **число** в `TELEGRAM_CHAT_ID` = id **того** пользователя, который нажал **Start** у **этого же** бота (username см. в выводе smoke после getMe). Уточнить id: бот **@userinfobot** в Telegram → вставить число в `.env` → открыть **ссылку на бота из вывода smoke** → Start → снова `python src/tg_smoke.py`.

## Решение (заполняет Mechanic)

### Причина

- Ручные one-liner из доков легко ошибиться путём, куда вставлять `sys.path`, и не было **одной** команды из корня с тем же `load_config()`, что у радара.
- Импорт `config` раньше вызывал `load_config()` на уровне модуля — при битом `.env` падал любой импорт до понятного smoke.

### Что сделано

- Добавлен **`src/tg_smoke.py`**: загрузка `load_config()` → **getMe** → **getUpdates** (limit=5) → **sendMessage** тестовым текстом в `TELEGRAM_CHAT_ID`; подсказки в консоль при 401, `chat not found`, пустом getUpdates, блокировке бота; **токен в консоль не печатается**; после getMe печатается `https://t.me/…`; проверка, что `TELEGRAM_CHAT_ID` похож на число; расширенные подсказки при `chat not found`.
- **`python src/main.py --telegram-smoke`** — тот же сценарий без отдельного файла запуска.
- В **`config.py`** убран вызов `load_config()` при импорте модуля (остаётся только явный `load_config()` в `main` и smoke).

### Подтверждение владельца (2026-05-20)

- Владелец подтвердил в чате: цепочка **заработала** (`tg_smoke` / Telegram ок).
- Корневая путаница в сессии: **`TELEGRAM_BOT_TOKEN` относился к одному @-боту (по getMe), а `/start` и переписка шли другому** (другой username). После выравнивания токена и бота — `sendMessage` 200.

### Как проверить (пошагово для владельца, без кода)

**Шаг A — только Telegram (обязательно)**

1. Откройте в проводнике папку **корня репозитория** (где лежит `.env`).
2. Откройте файл **`.env`** в редакторе и мысленно сверьтесь с **чеклистом** выше (не присылайте содержимое в чат).
3. Откройте терминал (PowerShell) в этой папке и выполните **одну** команду:
   - `python src/tg_smoke.py`  
   **или** то же самое: `python src/main.py --telegram-smoke`
4. Дождитесь трёх шагов в выводе (`getMe` → `getUpdates` → `sendMessage`).
5. Успех: в конце написано, что **sendMessage ок**, и в Telegram пришло сообщение вида «FL Radar: тестовая отправка…». Код выхода в терминале **0** (в PowerShell: `$LASTEXITCODE` равен 0).
6. Если ошибка — прочитайте **красные** подсказки под шагом; чаще всего: новый токен в BotFather, `/start` этому боту с нужного аккаунта, поправить `TELEGRAM_CHAT_ID`. Повторите шаг 3.

**Шаг B — радар после успешного шага A**

1. Запустите радар как обычно: `python src/main.py`.
2. Дождитесь цикла (интервал из `POLL_INTERVAL_MINUTES` в `.env`, минимум 10 минут по ТЗ) **или** дождитесь новой карточки по фильтру — уведомление должно уйти тем же ботом в тот же чат.
3. При сбое смотрите `data/radar.log` (или путь из `RADAR_LOG_PATH`): в поле `err` уже есть `description` от API (см. закрытый тикет про лог).

### Внешний стопер (без правок кода)

Если после чеклиста и нового токена **getMe** всё ещё не «ок» — сеть/VPN/блокировка `api.telegram.org` на машине или корпоративный прокси. Тогда: другая сеть, проверка доступа к `https://api.telegram.org` в браузере, написать Lead **одной фразой** с симптомом сети.
