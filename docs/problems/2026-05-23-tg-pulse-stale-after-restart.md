# TG: нет пульса после перезапуска (ложный «завис»)

**Статус:** решено  
**Дата:** 2026-05-23

## Симптом

- В боте / пульте: `Telegram-монитор: 🔴 завис? (2204с без пульса)`
- Алерт: `+7XXXXXXXXX1 не отвечает` — `монитор не пульсирует (2094с без пульса)`
- Биржи работают (`карточки_fl=90`, цикл идёт)

## Факты из `data/radar.log`

- Последний **`тг:пульс`:** `2026-05-23 14:55:20`
- Потом **`тг:монитор:старт`** acc1/2/3 в `15:30:22` и `15:32:16`, но **новых `тг:пульс` нет**
- `здравье:сбой` в `15:30:14` — проверка **до** старта монитора (гонка при рестарте бирж)
- `data/.tg_main.lock` — файл занят (процесс или зомби-lock)

## Не баг (норма)

Строки цикла `skip:ai:МИМО` / `skip:budget` — фильтр и ИИ, **не ошибки** радара.

## Гипотезы для Mechanic

1. `tg_main` завис в `run_monitor()` (3 клиента) — heartbeat не пишет пульс
2. Зомби `.tg_main.lock` при падении процесса — health думает «монитор есть», пульса нет
3. Health check из `main.py` слишком рано после `тг:старт` — ложный алерт при рестарте

## Что проверить владельцу (до Mechanic)

1. Пульт **⏹ Стоп** или `scripts\stop-radar.bat`
2. Если TG не гасится — закрыть 3 python в диспетчере задач
3. Удалить `data\.tg_main.lock` **только если** python tg_main нет
4. Пульт **▶ Старт** → подождать **3 мин** → в логе должны быть `тг:пульс` каждые ~2 мин
5. **ℹ Статус** — `Telegram-монитор: 🟢 работает`

## Задача Mechanic

- [x] Почему нет `тг:пульс` после `тг:монитор:старт` (15:30+)
- [x] Не слать алерт «не пульсирует», если `тг:старт` < 5 мин назад (гонка)
- [x] При выходе пульта / stop — гарантированно kill TG + снять lock

## Решение

### Причина

1. **Старый пульс** в SQLite после рестарта: health/status сравнивали с меткой прошлого сеанса (14:55), пока новый heartbeat ждал **120 с** до первого `тг:пульс`.
2. **Гонка:** `main.py` вызывал health до первого пульса нового `tg_main` → алерт «не пульсирует».
3. **Lock:** при жёстком kill процесса lock-файл мог остаться до ручного удаления.

### Что сделано

1. **`тг:старт`:** сброс `tg_monitor_last_pulse`, запись `tg_monitor_started_at`, окно прогрева **5 мин** — без алерта и без «завис?» по старому пульсу.
2. **Heartbeat:** первый `тг:пульс` сразу после старта, далее каждые 120 с (раньше — только после первой паузы 120 с).
3. **Stop:** `tg_main` снимает lock в `finally`; пульт и `stop-radar.bat` вызывают `try_release_stale_tg_main_lock()` после kill процессов.

### Изменённые файлы

- `src/health_check.py`
- `src/radar_status.py`
- `scripts/tg_main.py`
- `scripts/radar_desktop.py`
- `scripts/stop-radar.bat`

### Как проверить

1. `scripts\stop-radar.bat` → убедиться, что нет `data\.tg_main.lock` (или файл есть, но второй tg_main стартует).
2. Пульт **▶ Старт** (или `start-radar-all.bat`).
3. В `data/radar.log` в течение **~30 с** после `тг:старт` — строка `тг:пульс`; далее каждые ~2 мин.
4. **ℹ Статус** в боте — `Telegram-монитор: 🟢` (прогрев или «пульс Nс назад»), без «завис?» сразу после старта.
5. В логе нет `здравье:сбой … не пульсирует` в первые 5 мин после `тг:старт` (допустимо `здравье:ок … прогрев`).
