# Радар: токен в логе + нечитаемые ошибки Telegram (HTTP 400)

**Статус:** решено (2026-05-20)

## Симптом

- В `data/radar.log` в поле `err` в тексте ошибок Telegram попадает **полный URL** вида `/bot<TOKEN>/sendMessage` (утечка токена в лог при сетевых ошибках `requests`).
- При ответе Telegram **HTTP 400** в лог попадает только общая фраза **«Telegram Bot API вернул HTTP 400»** — **нет** поля **`description`** из JSON ответа API (владелец не видит, например, `chat not found`).
- Нет строки в логе **в самом начале** цикла до сетевых вызовов — владелец не понимает, «жив» ли процесс, пока не закончится круг.

## Ожидалось

- По тикету Lead / критерию в `docs/team/TASKS.md` («В работе» — лог и безопасность TG): старт цикла в логе до FL/TG; маскирование токена в любых сообщениях, попадающих в лог; при ошибках API — текст **`description`** из ответа Telegram, **без** токена в строке.

## Фактически

- Примеры из `data/radar.log` (фрагменты, **токен замазан**):  
  `… Max retries exceeded with url: /bot***MASKED***/sendMessage …`  
  `… Telegram Bot API вернул HTTP 400.` (без деталей `description`)

## Контекст

- Windows, запуск из корня: `python src/main.py`.
- Затронутые модули (ориентир): `src/main.py`, `src/telegram_notify.py`.
- Настройка чата Telegram (`chat not found`, `/start` у бота) — **вне** этого тикета (владелец + `docs/ops/RUN.md` §7); тикет только про **код логирования и сообщений об ошибках**.

## Решение (заполняет Mechanic)

**Статус:** решено (2026-05-20)

### Причина

1. В тексте `requests.RequestException` urllib3 подставляет URL запроса; для Bot API в пути есть `/bot<token>/`, поэтому токен оказывался в поле `err` лога при обёртке исключения в `TelegramNotifyError` и `_short_err`.
2. При `resp.status_code != 200` ответ не разбирался: сообщение об ошибке не дополнялось полем `description` из JSON Telegram.
3. Первая запись в лог за цикл делалась только в `_log_cycle` в конце (или при раннем выходе после fetch), без явной отметки старта до сетевых вызовов.

### Что сделано

- В `main.py`: перед `fetch_listing_projects` пишется строка `{ts} cycle_start` в тот же `radar.log`.
- В `main.py`: перед усечением в `_short_err` применяется замена фрагмента пути `/bot…/` на `/bot***MASKED***/`, чтобы токен не попадал в `err` ни из TG, ни из других исключений с таким URL.
- В `telegram_notify.py`: при HTTP ≠ 200 после `resp.json()` в сообщение об ошибке добавляется `description` из тела ответа API (если есть).

### Изменённые файлы

- `src/main.py`
- `src/telegram_notify.py`
- `docs/problems/2026-05-20-radar-log-token-leak-and-telegram-400.md` (этот тикет)

### Как проверить

1. Запуск из корня: `python src/main.py`.
2. В `data/radar.log` (или путь из `RADAR_LOG_PATH`): перед строкой `cards=…` той же метки времени есть строка `…Z cycle_start`.
3. Смоделировать сетевой сбой к Telegram или ответ с ошибкой: в поле `err` нет незамаскированного `/bot<секрет>/`; при HTTP 400 от Bot API в тексте ошибки видно `description` (например, про чат), без полного URL с токеном.
